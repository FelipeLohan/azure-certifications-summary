
## O que é?
Distribui o tráfego de rede de entrada entre múltiplas instâncias de VMs (ou outros recursos) para garantir **alta disponibilidade e escalabilidade**.

## Como funciona
1. O cliente faz uma requisição para o **IP público do Load Balancer**
2. O Load Balancer aplica um **algoritmo** para escolher qual VM responderá
3. A requisição é encaminhada para a VM selecionada
4. A VM processa e devolve a resposta

## Algoritmo padrão
**Hash de 5 tuplas:** IP origem + IP destino + Porta origem + Porta destino + Protocolo.
Garante que a mesma sessão de um cliente vá sempre para a mesma VM (session affinity básica).

## Tipos de Load Balancer

| | Load Balancer (Camada 4) | Application Gateway (Camada 7) |
|---|---|---|
| Protocolo | TCP / UDP | HTTP / HTTPS |
| Roteamento | Por IP e porta | Por URL, cabeçalho, cookie |
| SSL termination | Não | Sim |
| WAF | Não | Sim (SKU WAF) |

## SKUs do Azure Load Balancer

| SKU | Características |
|---|---|
| **Basic** | Gratuito, sem SLA, até 300 instâncias, sem Availability Zones |
| **Standard** | Pago, SLA 99,99%, Availability Zones, recomendado para produção |

## Componentes principais
- **Frontend IP:** endereço IP que os clientes acessam
- **Backend Pool:** conjunto de VMs que recebem o tráfego
- **Health Probe:** verifica se as VMs estão saudáveis (se não responde, é removida)
- **Load Balancing Rule:** define como mapear porta do frontend para o backend
- **Inbound NAT Rule:** encaminha tráfego para uma VM específica (ex: acesso SSH/RDP individual)

## SKU Standard — recursos extras
- Suporte a **Availability Zones**
- **Outbound rules** (controle do tráfego de saída das VMs)
- Integração com **Azure Monitor**

## Ponto-chave para o exame
> O Load Balancer opera na **Camada 4 (transporte)** — TCP/UDP. Não entende HTTP, URLs ou cookies. Para isso, use o **Application Gateway**.
